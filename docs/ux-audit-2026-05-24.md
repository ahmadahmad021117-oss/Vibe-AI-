# VibeCal — UX / UI Debug Session

**Build audited:** Debug, `com.vibecal.app`, simulator iPhone 17 Pro Max (iOS 26.3)
**Method:** Manual walkthrough of every reachable screen + source review for non-reachable states (camera preview, error states).
**Severity scale:** Critical = blocks task / risks App Store rejection · Major = wrong information shown to user · Minor = friction / polish · Cosmetic = small visual nit.

---

## P0 — Critical (fix before ship)

### C1. Paywall has no Privacy Policy / Terms of Service links
- **Where:** [PaywallView.swift:235-240](VibeNutrition/Features/Paywall/PaywallView.swift:235), `finePrint`.
- **What:** The fine print only discloses auto-renewal. App Review guideline **3.1.2(a)** requires functional links to the Privacy Policy and EULA/Terms on every auto-renewing-subscription screen.
- **Fix:** Add two tappable links beside (or under) `finePrint` that present `PrivacyPolicyView()` / `TermsOfServiceView()` (already implemented under `Features/Legal`).

### C2. Goal-weight onboarding lets you "Gain weight" with goal = current
- **Where:** [WeightScreens.swift](VibeNutrition/Features/Onboarding/Screens/WeightScreens.swift) + plan generator.
- **What:** Picked **Gain weight** → both weight screens default to 70.0 kg → Continue is enabled → Plan Preview shows **"Calibrated for lean weight gain."** but Weekly projection is **0.00 kg/week**, "Goal adjustment +0%", and Projection card on Progress says **"You're already at your goal."**
- **User impact:** First-run users finish onboarding believing the app is broken (math says "do nothing" while subtitle promises gains).
- **Fix:**
  1. After Goal is chosen, set the goal-weight slider default to `currentWeight ± reasonable delta` (e.g. +5 kg for Gain, −5 kg for Lose, =current only for Maintain).
  2. Block Continue when `(goal == current && goalType != .maintain)` with inline helper ("Pick a target above your current weight").

### C3. Camera screen has no way to dismiss while loading
- **Where:** [CameraScreen.swift:17-24](VibeNutrition/Features/Scan/CameraScreen.swift:17). `topBar` (with the X close button) is only mounted when `camera.isReady`.
- **What:** Tap Scan → permission granted → black screen with spinner. There is **no close button, no back gesture, no system swipe-down** (it's a full-screen cover, not a sheet). User must force-quit or wait.
- **Fix:** Move `topBar` outside the `isReady` branch so the X is rendered during the loading and permission-denied states too.

## P1 — Major (wrong / misleading information)

### M1. Macros carousel formats decimals inconsistently
- **Where:** [DashboardView.swift:252,256,259](VibeNutrition/Features/Dashboard/DashboardView.swift:252) — `fractionDigits` is set to 1 for Iron/Zinc but defaults to 0 for Magnesium. Same pattern on the Vitamins card.
- **Visible as:** "Magnesium 0 / 420 mg" next to "Iron 0.0 / 8.0 mg" and "Zinc 0.0 / 11.0 mg" — three rows in the same card, three different number styles.
- **Fix:** Pick one rule per card (suggest: `digits = (target < 10) ? 1 : 0`) and apply everywhere.

### M2. Number formatting uses space-as-thousands across the app
- **Where:** Calorie ring on Dashboard ("4 026"), Plan Preview ("2 610"), TDEE rows. This is the French / SI locale separator.
- **What:** App is shipped in `en.lproj` (English). English users read "2 610" as a typo or as 2/610. Probably a missing `numberStyle = .decimal` with explicit `Locale(identifier:)`.
- **Fix:** Use `value.formatted(.number)` with explicit US locale, or `NumberFormatter` with `usesGroupingSeparator = true` and `locale = Locale.current` only after the in-app unit toggle is wired up. Or accept it as locale-aware but verify against device locale, not French default.

### M3. Onboarding pace screen shows "Maintain · Balanced" even when user picked Gain weight
- **Where:** PaceScreen / projection card. Symptom of C2 (zero-delta), but the *label string* of the projection card should at minimum reflect the goal the user picked, not the math output. Today the label `Maintain` is being derived from `delta == 0`.
- **Fix:** Drive the label off the goal enum, not off the computed delta. Show a warning state ("Pick a target weight to see your projection") when delta is zero.

### M4. Pre-selection contradicts the "Recommended" badge on Apple Health step
- **Where:** [HealthSyncScreen.swift](VibeNutrition/Features/Onboarding/Screens/HealthSyncScreen.swift).
- **What:** "Connect Apple Health" is labeled **Recommended**, but **"Skip for now" is the option that is pre-selected** with a checkmark. The user reads "Recommended" → looks for the check on it → confused.
- **Fix:** Either pre-select the Recommended option (default behavior) or remove the "Recommended" tag.

### M5. Dietary preferences is single-select
- **Where:** [DietaryPrefScreen.swift](VibeNutrition/Features/Onboarding/Screens/DietaryPrefScreen.swift).
- **What:** A user who is Halal + High-protein, or Vegan + Keto, cannot represent themselves. Choices are mutually exclusive but the real world isn't.
- **Fix:** Convert to multi-select chips. Add the common missing options (Gluten-free, Dairy-free, Pescatarian, Allergies sub-step).

### M6. Sign-in falls back to silent sign-up on wrong password
- **Where:** [AuthGateView.swift:154-171](VibeNutrition/Features/Auth/AuthGateView.swift:154).
  ```
  catch { do { signUp; signIn } catch { surface } }
  ```
- **What:** A returning user who mistypes their password gets a brand-new account created on their behalf (or hits "Email already in use" with no path to reset). Real "wrong password" is masked.
- **Fix:** Distinguish "user-not-found" (then sign up) from "invalid-credentials" (then show "Wrong password. Forgot it?" with a reset link).

### M7. "Late night" header is judgmental — and runs 22:00–05:00 (7h of the day)
- **Where:** [DashboardView.swift:402](VibeNutrition/Features/Dashboard/DashboardView.swift:402).
- **What:** Many fitness-app users log dinner at 22:30. Greeting them with "Late night" reads as a soft "you're doing it wrong." For a "Gain weight" user trying to hit a 2 610 kcal target this is a daily friction point.
- **Fix:** Use a neutral late-night greeting ("Evening", "Still up?", or just no greeting after 22:00).

### M8. "Today" empty-state copy is clipped behind the tab bar
- **Where:** Dashboard, post-onboarding fresh state. Sparkles icon + "Snap a m… seconds" text is hidden by the floating TabView.
- **Fix:** Increase the trailing `Spacer(minLength:)` (currently 80) to ~120–140 to clear the tab bar, or pad the ScrollView's bottom safeAreaInset to match the tab bar height.

## P2 — Minor (friction / consistency)

### m1. Continue-button enablement is inconsistent across onboarding
- **Age screen** keeps Continue disabled until the user taps +/− (even though "22" is shown). **Height / Training days / Meals per day** all accept the default and enable Continue immediately.
- **Fix:** Pick one. Recommendation: every numeric onboarding step should accept the default → Continue is always enabled.

### m2. No keyboard / direct-input for any numeric onboarding step
- Weight, goal weight, height, age, meals per day, training days are all slider- or stepper-only. To land on 75.5 kg or 178 cm precisely you'd be dragging a slider through ~150 detents.
- **Fix:** Make the number tappable → opens a numeric keypad sheet that writes back into the same field. Common iOS pattern (used by Apple Fitness, Cronometer, MFP).

### m3. Goal screen + Main focus screen are near-duplicates
- Goal asks: Lose / Gain / Build muscle / Maintain / Recomposition / Improve health (6 options).
- Main focus asks: Fat loss / Muscle gain / Recomposition / General health (4 options).
- User can't tell the difference between "Build muscle" and "Muscle gain", or "Improve health" and "General health".
- **Fix:** Merge into one step, or rename Main focus to "Calorie aggressiveness" with three labels that don't overlap the goal vocabulary (e.g. "Gentle / Moderate / Aggressive").

### m4. Manual entry sheet has no placeholders and doesn't auto-compute kcal
- **Where:** [ManualEntrySheet.swift](VibeNutrition/Features/Dashboard/ManualEntrySheet.swift).
- Fields Name / kcal / Protein / Carbs / Fat all show blank. The standard nutrition relationship `kcal ≈ 4P + 4C + 9F` is not exposed — if the user fills in macros the kcal field stays blank and Save is disabled.
- **Fix:** Add placeholders ("e.g. Chicken & rice"), and auto-fill kcal from macros (let user override).

### m5. Weight check-in has no keyboard or date picker
- **Where:** [WeightCheckInSheet.swift](VibeNutrition/Features/Dashboard/WeightCheckInSheet.swift) / [EditWeightSheet.swift](VibeNutrition/Features/Profile/EditWeightSheet.swift).
- Slider-only entry, today-only timestamp. Users who weighed yesterday and forgot can't backdate the entry; users who weigh themselves on a non-round number (75.3 kg) can't easily land it.
- **Fix:** Add tap-the-number-to-type, and a small date selector ("Today / Yesterday / Pick…").

### m6. "Sign out" row uses a chevron and default text color
- **Where:** [ProfileView.swift:125-138](VibeNutrition/Features/Profile/ProfileView.swift:125).
- The icon is red but the label is the standard `Theme.Palette.text`, and the row uses the same chevron as "Marketing email". Sign-out is destructive (or at least not navigational); it shouldn't visually equate to a sub-page.
- **Fix:** Drop the chevron and tint the label `Theme.Palette.danger` (the Delete row already does this).

### m7. Paywall plan comparison is uneven
- "Premium Weekly $6.99" vs "Premium Yearly $39.99 · $0.77 / week".
- Weekly row hides the equivalent per-day cost; yearly row brags the per-week. Side-by-side feels asymmetric.
- **Fix:** Show both rows in the same form: "$6.99 / week" and "$39.99 / year · $0.77 / week".

### m8. Paywall CTA hides the post-trial price
- "Start free trial" without an inline reminder of "then $0.77/week". Some App Reviewers flag this; conversion testing usually likes the price + trial framed together.

### m9. Auth screen has no Privacy / Terms disclosure
- Sign in with Apple + Continue with email at the bottom, no "By continuing, you agree to…" line. This is the same App-Store guideline that bites the paywall (C1) — but the auth screen is where the account is created, so it's the earlier checkpoint.

### m10. Projection chart axes are weakly labeled
- Y-axis displays "70" three times (top, middle, bottom). X-axis is "0w" with no end label.
- **Fix:** Show real min/max on Y, and a clear "Today → May 31, 2026" on X.

### m11. "= 0.0 kg" badge is opaque
- The leading `=` is not a familiar UX glyph. Users read it as "equals … what?"
- **Fix:** Replace with explicit copy: "0 kg to go" / "+5 kg to go" / "On target."

### m12. Macros carousel page-indicator dots aren't reachable by keyboard / VoiceOver — they're decorative on a `TabView(.page)` style. Accessibility audit needed.

## P3 — Cosmetic / polish

- **p1.** Profile sheet title in the rendered build reads **"Profile"** but the source ([ProfileView.swift:88](VibeNutrition/Features/Profile/ProfileView.swift:88)) says **"Settings"**. Either the running build is stale or there are two profile entry points. Confirm which one ships.
- **p2.** Onboarding progress bar at the top has no "Step X of Y" label — users can't tell how long the flow is. (~13 screens.)
- **p3.** Age and Meals-per-day screens leave huge vertical whitespace between title and stepper; consider tightening (lift the stepper closer to the title or add an illustration to fill the gap).
- **p4.** Camera screen has only two real controls (library, shutter) and an empty third slot on the right. Either fill it (flash toggle) or remove the placeholder to re-center the shutter.
- **p5.** Auth tagline "AI calorie tracking that actually clicks." — "clicks" is a verb pun that could read as button-click or click-track. A clearer tag like "Calorie tracking that finally sticks." removes the ambiguity.
- **p6.** Hero in onboarding: lots of empty space between the H1 title and the picker on Age / Training-days / Meals — these could converge to a single uniform layout.
- **p7.** Status-bar carry-over of "◄ VibeCal" appears because the app was launched via `simctl launch` from terminal. Not a bug; will not happen in real-world install.

## What works well (don't change)

- **Sex-assigned-at-birth step** — inclusive ("Prefer not to say" with a sensible fallback), explains *why* it's asked, mentions privacy. Best onboarding step in the app.
- **Plan Preview** transparency block ("How we got there" with BMR, multiplier, TDEE, adjustment) is excellent for trust. Most calorie apps hide the math.
- **Streak pill** logic ([DashboardView.swift:105-110](VibeNutrition/Features/Dashboard/DashboardView.swift:105)) — the comment shows real product thinking ("muted pill made a 14-day streak look identical to day 1").
- **Camera permission-denied view** is well designed — alternative path (photo library) is the most prominent CTA.
- **Paywall benefit copy** is outcome-focused not feature-focused ("never wonder what to eat", "see your weekly wins"), good for conversion.
- **Selected-state styling** across the app (mint border + check) is crisp and consistent.
